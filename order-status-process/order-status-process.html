The Order Status Process

How a supplier restock travels from order → delivery → goods-in → marked off — and what the system does (and quietly doesn't do) when it never arrives.

Scope supplier restock lifecycle + non-delivery Source reverse-engineered from code Audience internal technical team

"We order a restock for a shoe and it doesn't arrive because the supplier was out of stock. What happens?"

The watched record — the orderstatus row and PowerBuilder's "on order" total — sits untouched, suppresses a reorder while it waits, then is silently purged at 30 days. No alert, no reorder trigger. The two durable records survive it (stockorder says we ordered; incoming_stock shows nothing arrived) — but nothing reconciles them automatically. That's the gap.

01 The core problem

One table, three unrelated jobs

orderstatus is keyed only by (ordernum, shopifysku) and split solely by ordertype. It isn't an order ledger — it's a transient work queue that gets purged on timers. That's why "did the restock arrive?" has no durable answer in this table.

03 Goods-in on scan

What the scanner decides, per unit

of_scan() resolves the barcode/FNSKU/SKU to skumap.code, then matches arrived=0 rows Amazon-first, then Local (first hit wins). The save routine then routes exactly one unit — always qty 1:

1 customernone — pick already aligned——Pick
3 Amazoninsert #FREEC3-AmazonamzC3-Amazon

Every branch also appends one row to incoming_stock (qty 1) and logs bclog "Goods In {code} → {target}". The type-1 customer branch is commented out in of_scan — the scanner no longer touches customer picks.

EDGE

Amazon-after-purge downgrade. If a type-3 row was purged (>30d) before a late delivery, the scan finds no match → the unit saves as unallocated local, not C3-Amazon/amz. Stock is banked; FBA intent silently lost. This is precisely the historic #FREE-leak failure mode.

EDGE

Partials can't show in orderstatus. arrived is a binary flag flipped on the first unit scanned. Order 4, receive 3 → the row still reads "arrived." The accurate count lives in incoming_stock (one row per unit), never in orderstatus.

04 Non-delivery

Where the three records stand when the shoe never comes
stockorder
ordered · kept 365d
Still says we ordered it. Durable for a year — well past every orderstatus purge.
incoming_stock
received · kept 3yr
No row for this code. Nothing was scanned, so nothing was ever appended.
orderstatus
transient · purged 30d
Gone. Inflated "on order" (suppressing reorder) until it silently vanished at day 30.
The detector is stockorder (ordered) vs incoming_stock (received) — the intelligence exists and survives the purge. But nothing compares them automatically; no standing reconciliation report exists (one was considered and declined as too heavy). Detection today = run stuck_supplier_orders.sql inside the 12–30d window, or notice the SKU is short later.
